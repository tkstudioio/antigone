# Conventional Commit Message Generator

## Overview

Every commit must be **standalone** and group **coherent files**. Message must follow [Conventional Commits v1.0.0](https://www.conventionalcommits.org/en/v1.0.0/).

Standalone = commit compiles, tests pass, no half-work. Coherent = all files serve one intent (one feature, one fix, one refactor). Never mix unrelated changes.

## ABSOLUTE PROHIBITION — AI Co-Author References

**The commit message MUST NEVER, under any circumstance, for any reason, contain references co-authoring an AI agent.**

This rule is absolute and non-negotiable. Violation = commit MUST be aborted and rewritten before any `git commit` runs.

Forbidden in subject, body, AND footer — including but not limited to:

- `Co-Authored-By: Claude <noreply@anthropic.com>`
- `Co-Authored-By: Claude Opus ...`
- `Co-Authored-By: GPT-...`, `Copilot`, `Cursor`, `Gemini`, `Codex`, any LLM/agent identity
- `Generated with Claude Code` / `🤖 Generated with ...` trailers
- Any `Co-Authored-By:` whose email domain is `anthropic.com`, `openai.com`, `google.com` (when referring to model), or any AI vendor
- Any prose like "written by AI", "generated by assistant", "with help from Claude"

Human co-authors (real teammates) are allowed. AI agents are not.

**Self-check before commit:** grep the final message for `Co-Authored-By`, `Generated with`, `Claude`, `GPT`, `Copilot`, `🤖`. Any hit → strip it → re-verify → only then commit.

## When to Use

- Running `git commit` or preparing commits
- Splitting dirty working tree into logical commits
- Reviewing staged files before commit
- Generating PR titles (same format)

## Structure

```
<type>[optional scope][!]: <description>

[optional body]

[optional footer(s)]
```

- **type**: required, lowercase
- **scope**: optional noun in parens describing codebase section
- **!**: optional, marks breaking change
- **description**: imperative mood, lowercase start, no trailing period, ≤72 chars
- **body**: blank line after description, explains _why_, wrap ~72 chars
- **footer**: blank line after body, `Token: value` or `Token #value`

## Types

| Type       | When                                   |
| ---------- | -------------------------------------- |
| `feat`     | new feature (MINOR semver)             |
| `fix`      | bug fix (PATCH semver)                 |
| `docs`     | documentation only                     |
| `style`    | formatting, whitespace, no code change |
| `refactor` | code change, no feat/fix               |
| `perf`     | performance improvement                |
| `test`     | add/fix tests                          |
| `build`    | build system, deps                     |
| `ci`       | CI config                              |
| `chore`    | maintenance, no src/test change        |
| `revert`   | revert prior commit                    |

## Breaking Changes

Two equivalent ways (use one or both):

1. `!` before colon: `feat(api)!: remove deprecated endpoint`
2. Footer: `BREAKING CHANGE: <what broke and migration>` (uppercase, always)

## Coherence Rules

Before writing message, inspect staged files:

1. **One intent per commit.** If files answer different "why", split.
2. **Standalone.** Commit must not break build. No "WIP", no "part 1".
3. **No mixing.** Don't combine `feat` + unrelated `fix` + `chore`. Split.
4. **Co-located changes OK.** Test file + impl file + types = one commit.
5. **Generated files** (lockfiles, migrations, snapshots) ride with change that caused them.

If staged tree fails coherence check → `git reset` and re-stage per intent.

## Workflow

```
1. git status + git diff --staged
2. Group files by intent (feature? fix? refactor?)
3. If >1 intent → reset, stage subset, commit, repeat
4. Pick type + scope from dominant intent
5. Write description (imperative, ≤72 char)
6. Add body if "why" non-obvious
7. Add footer for breaking/refs/co-authors
```

## Examples

**Simple feat:**

```
feat(auth): add JWT refresh endpoint
```

**Fix with scope + body:**

```
fix(parser): handle empty input array

Previously threw TypeError on []. Now returns [] early.
```

**Breaking via !:**

```
feat(api)!: drop v1 response envelope

BREAKING CHANGE: clients must read `data` at root, not `body.data`.
```

**With footers (human co-author OK):**

```
fix(db): prevent stale connection reuse

Closes #423
Reviewed-by: alice
Co-Authored-By: Bob Real <bob@company.com>
```

**Revert:**

```
revert: feat(auth): add JWT refresh endpoint

Refs: a1b2c3d
```

## Common Mistakes

| Wrong                                  | Right                                               | Why                              |
| -------------------------------------- | --------------------------------------------------- | -------------------------------- |
| `Fixed bug`                            | `fix: correct null check in parser`                 | type required, imperative        |
| `feat: Added login.`                   | `feat: add login`                                   | lowercase, no period, imperative |
| `feat(Auth): ...`                      | `feat(auth): ...`                                   | scope lowercase convention       |
| `chore: update deps and fix login bug` | split into `chore(deps): bump X` + `fix(auth): ...` | two intents                      |
| `feat!: new API` (no explanation)      | add `BREAKING CHANGE:` footer                       | `!` alone hides migration info   |
| `wip: stuff`                           | don't commit                                        | not standalone                   |

## Red Flags — Stop and Split

- Message uses "and" to join intents → split
- Staged diff touches unrelated modules → split
- "Also fixed..." in body → separate commit
- Description >72 chars → too broad, split or rewrite
- Can't pick one type → multiple intents, split

## Quick Reference

```
feat(scope): add X           → new feature
fix(scope): correct X        → bug fix
refactor(scope): extract X   → no behavior change
perf(scope): speed up X      → faster, same behavior
docs(scope): clarify X       → docs only
test(scope): cover X         → tests only
chore(deps): bump X          → maintenance
feat(scope)!: remove X       → breaking
```

Description verbs: `add`, `remove`, `fix`, `correct`, `rename`, `extract`, `inline`, `bump`, `migrate`, `introduce`, `drop`.

## Output Contract

When generating message for agent caller, return ONLY the commit message (no prose, no fences). Caller pipes to `git commit -m` or HEREDOC.
